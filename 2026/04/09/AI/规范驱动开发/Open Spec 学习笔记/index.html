SDD 与 Open Spec 基础

SDD 规范驱动开发

SDD（Spec-Driven Development，规范驱动开发）是 AI 时代的一种新型开发方法论，强调在编码之前先创建明确的、AI 能直接理解和执行的规范文档

传统开发流程是：想到什么写什么，边写边改，最后再补文档。这样容易导致需求不清晰、代码和文档对不上

而 SDD 的思路正好相反：先把需求写成规范文档，并且把规范文档当作代码的唯一真相来源

你可以把规范文档理解成 “项目宪法”，它包含了详细的需求描述、系统设计和接口定义。AI 必须严格遵守这些条文来生成代码，确保产出完全符合预期


为什么 SDD 越来越受重视

因为 AI 生成代码的质量直接取决于上下文的清晰度，而不仅仅是依靠提示词技巧。一个清晰的规范文档能比任何 Prompt 黑魔法更有效地减少错误

SDD 的典型工作流程如下：

  1. Constitution 制定准则：定义项目的基本原则、代码规范、性能标准
  2. Specify 编写规范：描述要做什么功能、为什么做、用户需求是什么
  3. Clarify 澄清疑问：让 AI 提出结构化问题，明确边界情况和错误处理
  4. Plan 制定方案：确定技术栈、系统架构、数据模型、API 接口
  5. Tasks 拆解任务：把计划拆解成可执行的任务列表，标注依赖关系和优先级
  6. Implement 执行实现：AI 按照任务列表生成代码，人类验证

其实这和程序员在企业中开发项目的标准流程非常相似，只不过执行者从人变成了 AI

Open Spec 简介

Open Spec 是一个开源的、轻量级的 SDD 框架，专门为 AI 编程助手设计，用来让你和 AI 在写代码之前先就 “要做什么” 达成一致

它由 Fission AI 团队在 2025 年 8 月开源，托管在 GitHub 上，采用 MIT 协议，目前已经积累了 6 万多 Star，是当前 Vibe Coding 领域最受欢迎的规范框架之一

Open Spec 的核心思路很简单：在项目里加一层轻量的规范文件，让人类和 AI 都能读到同一份 “要做什么” 的说明，避免需求只活在聊天记录里

打个比方，以前你和 AI 对话就像两个人口头交代任务，说完就忘了；有了 Open Spec，相当于把任务写成了一份正式的工单贴在墙上，谁都看得见，谁都不能赖账

为什么需要 Open Spec

AI 编程助手虽然强大，但有一个致命问题：当需求只存在于聊天记录里时，AI 的表现是不可预测的

AI 是一个非常自信的 “实习生”。你给它一句话需求，它会立刻动手，但中间有任何没说清楚的地方，它就会自己脑补，而且补得特别自信。等你发现做偏了，代码已经写了一大堆，回炉重来的成本很高

Open Spec 把 “达成一致” 这个动作提前到了写代码之前。在这个阶段，改一行规范文档的成本几乎为零；而等代码写完再改，成本可能是几十倍

所以 Open Spec 解决的不是 “AI 不会写代码” 的问题，而是 “AI 写的不是你想要的” 的问题。它把分歧消灭在萌芽阶段，而不是等成果出来再返工

Open Spec 的设计哲学

Open Spec 有五条非常清晰的设计原则，理解了这五条，就理解了它为什么这么设计：

  • fluid not rigid：流畅而非死板。没有强制的阶段门禁，你可以随时回头改前面的制品
  • iterative not waterfall：迭代而非瀑布。规范是边写边完善的，不是一次性定死
  • easy not complex：简单而非复杂。5 分钟就能上手，不需要装一堆依赖
  • built for brownfield not just greenfield：为既有项目而生，不只是新项目。这是 Open Spec 最区别于其他工具的一点
  • scalable from personal projects to enterprises：从个人项目到企业都能用

其中最值得关注的是 brownfield-first（既有项目优先）。市面上大多数 SDD 工具都假设你从零开始一个新项目，但现实里绝大多数人是在一个已经存在了很久、代码库已经很大的项目上加功能。Open Spec 不要求你一开始就把整个系统文档化，而是 “改到哪写到哪”，规范会随着你实际做的工作慢慢填充起来

核心概念

Spec 规范

Spec（规范）是 Open Spec 里描述系统行为的文档，用纯 Markdown 写成，不需要学任何特殊语法

一个 Spec 的结构非常朴素：它包含若干个 Requirement（需求），每个 Requirement 又包含若干个 Scenario（场景）

Spec 的核心思想是：它描述的是 “系统应该怎么表现”，而不是 “系统现在怎么实现”。代码会变，但规范相对稳定，所以规范是真相来源，代码是实现细节

Requirement 需求

Requirement（需求）是 Spec 里描述某一条具体系统行为的单位

写 Requirement 时，Open Spec 借用了 RFC 2119 的关键词来表明需求的强度：

  • MUST：必须做到，没有任何商量余地
  • SHALL：应当做到，是规范层面的强制要求
  • SHOULD：建议做到，但有例外情况可以不这么做
  • MAY：可以选择做，是可选能力

这些关键词不是装饰，它们让 AI 能准确判断一条需求的优先级。比如 “The system MUST require 2FA” 和 “The system MAY support 2FA” 在 AI 眼里是完全不同的指令

写好 Requirement 的关键是要具体、可验证。不要写 “系统应该很好用” 这种模糊的话，而要写 “系统 SHALL 在用户点击提交后 200ms 内给出反馈”

Scenario 场景

Scenario（场景）是 Requirement 的具体例子，描述某条需求在实际中是怎么运作的

Open Spec 推荐用 GIVEN/WHEN/THEN 的格式来写场景，这是行为驱动开发（BDD）里非常成熟的写法：

  • GIVEN：给定一个初始状态
  • WHEN：当某个动作发生
  • THEN：那么系统应该有什么样的结果

Delta Spec 增量规范

Delta Spec（增量规范）是 Open Spec 最核心、也最巧妙的概念

很多 SDD 工具的做法是：每次改功能都要重写一遍完整的规范。这在既有项目上几乎不可能落地，因为没人愿意为了加一个暗色模式去重写整个系统的文档

Open Spec 的做法是 Delta Spec：它只描述 “相对于当前规范，这次改了什么”，用三个段落来组织：

  • ADDED：新增了哪些需求
  • MODIFIED：修改了哪些已有需求
  • REMOVED：删除了哪些需求

这种设计让既有项目的改造成本极低。你不需要一开始就写全整个系统的规范，只需要在每次改动时写一份 Delta，改完归档时再合并进主规范。规范库会随着你的工作自然生长

Change 变更

Change（变更）是 Open Spec 里一次修改的完整包装。每次你想给系统加功能、改行为、删东西，都对应一个 Change

一个 Change 在文件系统里就是一个独立的文件夹，里面装着这次修改涉及的所有制品：提案、增量规范、设计文档、任务清单

这样做的好处是：一次修改的所有上下文都集中在一起，不会散落在聊天记录里。你可以随时翻回去看 “当时这个功能是怎么想的”，也可以把一个 Change 交给同事或另一个 AI 审查

当 Change 完成后，它会被归档（Archive），它的 Delta Spec 会合并进主规范库，文件夹本身会被移到 archive 目录里留档

Artifact 制品

Artifact（制品）是 Change 文件夹里的一份份文档。Open Spec 默认有四类标准制品：

制品 作用
proposal.md “为什么做” 和 “做什么”，记录意图、范围、方案
specs/ 增量规范，描述需求的变化
design.md “怎么做”，技术方案和架构决策
tasks.md 实现清单，带复选框的任务列表

这四类制品是有依赖关系的：proposal 决定 specs，specs 决定 design，design 决定 tasks，tasks 驱动 implement

但这个依赖不是单向的水流。Open Spec 强调 “边做边学”，你在实现阶段发现设计要改，完全可以回头去改 design 甚至 proposal，没有强制的阶段门禁

Domain 领域

Domain（领域）是 Open Spec 组织规范的方式。规范不是一锅粥堆在一起，而是按业务领域分文件夹存放

比如一个电商系统，规范目录可能是这样的：

每个领域一个文件夹，里面放该领域的 spec.md。这种组织方式让规范库天然和你的业务边界对齐，找一个功能该看哪份规范一目了然

Schema 模式

Schema（模式）是 Open Spec 里定义 “有哪些制品类型、它们之间怎么依赖” 的元数据

默认的 Schema 叫 spec-driven，包含前面提到的四类标准制品。如果你觉得默认工作流不适合自己，可以 fork 出一份自定义 Schema：

然后编辑它，比如加一类 “impact.md” 影响分析文档，或者去掉 design.md。这让 Open Spec 既能开箱即用，又能适配各种团队的特殊流程

Source of Truth 真相来源

Source of Truth（真相来源）在 Open Spec 里特指 openspec/specs/ 目录，里面存放的是当前系统 “已经达成一致的、正在生效的” 行为规范

整个 Open Spec 的工作流可以理解成一个 “真相来源的维护机制”：

  • 你想改东西，先写一份 Delta（提案中的增量规范）
  • 大家审核通过后才开始写代码
  • 代码写完、验证通过，Delta 就合并进主 specs，成为新的真相

这样无论何时，任何人或 AI 想知道 “系统现在应该怎么表现”，去读 specs/ 就够了，不需要去翻代码、翻聊天记录、翻 wiki。规范库永远是最新、最权威的那一份

项目结构

openspec 目录结构

运行 openspec init 之后，你的项目里会多出一个 openspec/ 目录，整个 Open Spec 的所有内容都住在这里

把规范放在代码仓库里、和代码一起用 Git 管理，是 Open Spec 一个非常重要的设计。这样规范就跟着代码走，不会出现 “代码在 GitHub，规范在某个文档平台，两边各活各的” 这种脱节情况

specs 目录

specs/ 是整个项目的真相来源。它描述的是系统 “当前应该怎么表现”，而不是 “将来想怎么表现”

这个目录是按领域（Domain）组织的，每个领域一个文件夹，里面放一份 spec.md。随着你做的 Change 越来越多，这个目录会自然地丰满起来

一个常见的误区是想一开始就把整个系统文档化。Open Spec 明确反对这种做法，它的建议是：只为你正在改动的部分写规范，规范库会随着实际工作慢慢长出来。这就像写测试一样，不要为了覆盖率而写测试，要为了验证行为而写测试

changes 目录

changes/ 目录里放的是正在进行中的变更。每个变更一个独立的文件夹，文件夹名就是变更名，比如 add-dark-mode、refactor-auth 这种

每个变更文件夹里装着这次修改的完整上下文：proposal、design、tasks、以及 delta specs。这样做的核心价值是 “上下文集中”——一次修改的所有思考过程都在一个地方，不会被聊天记录冲散

当变更完成并归档后，文件夹会被移到 changes/archive/ 下，并加上日期前缀，比如 2026-07-21-add-dark-mode。归档目录是审计历史，你可以随时翻回去看 “当时这个功能是怎么设计的”

config.yaml 配置文件

config.yaml 是 Open Spec 的项目级配置文件，是可选的。不写也能用，写了可以做一些定制化的事

常见的配置项包括：选择哪个工作流 profile（core 还是 expanded）、自定义 Schema、指定支持的 AI 工具等

如果你只是个人用默认工作流，完全可以不创建这个文件，Open Spec 会用合理的默认值

制品体系

proposal.md 提案

proposal.md 是一次变更的 “为什么” 和 “做什么”，是整个变更的起点

一份典型的 proposal 长这样：

proposal 一般包含三块内容：Intent（为什么做这个改动）、Scope（这次改动的范围有多大）、Approach（打算怎么做）

proposal 的价值在于逼你把 “我到底想干嘛” 想清楚。很多时候我们脑子里觉得需求很明确，一写下来才发现其实自己也没想明白。proposal 就是把模糊的想法变成清晰文字的第一道关

specs 增量规范

变更文件夹里的 specs/ 子目录放的是 Delta Spec，也就是这次变更会带来哪些需求上的变化

它的格式前面讲 Delta Spec 时已经介绍过，用 ADDED / MODIFIED / REMOVED 三段来组织。这里补充一点：Delta Spec 是给审查者看的，也是给 AI 看的

给审查者看：审查者不用读全部代码，只看 Delta 就能知道这次改动会怎么影响系统行为，判断是不是想要的

给 AI 看：AI 在实现时会拿 Delta Spec 当 “验收标准”，每写完一个功能就对照场景检查一下，是不是符合规范要求

design.md 设计文档

design.md 是一次变更的 “怎么做”，记录技术方案和架构决策

不是所有变更都需要 design.md。如果改动很简单（比如改个文案、调个颜色），proposal 里说两句就够了。但如果改动涉及架构选择、数据模型变化、跨模块影响，就需要一份 design 来讲清楚技术上的取舍

design.md 通常会回答这些问题：选了什么技术方案、为什么不选其他方案、会改哪些文件、有没有性能或安全影响

一个常见的好习惯是在 design 里记录 “考虑过但没选” 的方案。这样以后回头看，能知道当时为什么做了这个决定，而不是只看到最终结果

tasks.md 任务清单

tasks.md 是把 design 拆解成可执行的任务列表，带复选框

tasks.md 是 AI 实现（apply）阶段的直接输入。AI 会按这个清单一项一项地做，做完一项打个勾。这种带勾的清单有两个好处：一是进度一目了然，断了重连也能接着干；二是做完后可以拿这个清单当验收单，逐项确认

任务拆解的粒度很关键。太粗了 AI 容易跑偏，太细了又显得啰嗦。一般建议每个任务对应一个可独立验证的小功能点

工作流程

核心工作流

Open Spec 的默认工作流叫 core profile，包含五个斜杠命令，构成一个完整的闭环：

一句话概括就是：先想清楚（explore），再写提案（propose），然后实现（apply），同步进度（sync），最后归档（archive）

这里有一个最容易让新手踩坑的点：openspec 命令是在终端里跑的，而 /opsx: 命令是在 AI 助手的对话框里输入的。两者别搞混了，混了就跑不通

/opsx:explore 探索

/opsx:explore 是一个 “零成本思考伙伴”。当你只有一个模糊的想法、还不确定要做什么的时候，先用它

它会读取你的代码库，列出几种可能的方案，帮你把模糊的想法打磨成一个具体的计划。整个过程不会产生任何制品，也不会写任何代码，纯粹是 “想”

Open Spec 官方特别强调这是和 AI 协作时最值得养成的好习惯。因为 AI 是个特别自信的实习生，你给它一个模糊需求，它会立刻动手做，而且做的特别起劲——但很可能做的根本不是你想要的。explore 就是让 AI 在动手前先和你对齐方向

当然，如果你已经想得很清楚了，也可以跳过 explore 直接 propose

/opsx:propose 提案

/opsx:propose 是核心命令，用来生成一份完整的变更计划

用法很简单，在 AI 对话框里输入 /opsx:propose add-dark-mode，AI 就会：

  1. 读取你的代码库，理解现状
  2. 在 openspec/changes/add-dark-mode/ 下创建四个制品：proposal.md、specs/、design.md、tasks.md
  3. 让你审核

注意这一步只生成文档，不写代码。你要做的是仔细审核这四份制品，看看 AI 理解的需求和你想的是不是一致。如果不对，直接改文档，比改代码便宜得多

propose 阶段的审核是整个 SDD 流程里性价比最高的一步。在这里改一个错别字的成本是 1 秒，等代码写完再发现理解错了，返工成本可能是 1 小时

/opsx:apply 应用

/opsx:apply 是让 AI 按 tasks.md 的清单开始写代码

AI 会一项一项地实现任务，每完成一项就打个勾。实现过程中如果发现 design 需要调整，可以随时停下来改 design，再继续

apply 阶段建议保持上下文干净。开始前清一下对话历史，让 AI 专注于当前变更的制品，不要被之前的对话干扰。Open Spec 官方推荐用高推理能力的模型（比如 Codex 5.5、Claude Opus 4.7）来做 apply，因为实现阶段需要严谨

/opsx:sync 同步

/opsx:sync 用来把实现过程中的进度同步回 tasks.md

有时候你会手动改一些代码，或者 AI 在 apply 过程中偏离了原始任务清单。sync 就是让文档和实际进度重新对齐，避免 “文档说做完了，代码其实没做” 这种脱节

这是一个保持真相来源可靠性的维护命令，建议在长任务里定期跑一下

/opsx:archive 归档

/opsx:archive 是变更生命周期的最后一步。当一个变更完成、验证通过后，运行它：

  1. ADDED 的需求被追加到主 specs
  2. MODIFIED 的需求替换主 specs 里的旧版本
  3. REMOVED 的需求从主 specs 里删除
  4. 变更文件夹被移到 changes/archive/ 下，加上日期前缀

归档完成后，主规范库就是最新的真相来源，下一次变更又会基于这份新规范开始。整个系统的规范就这样随着工作自然演进

扩展工作流

除了默认的 core profile，Open Spec 还提供了一套扩展工作流命令，适合需要更细粒度控制的场景：

  • /opsx:new：手动创建一个空的变更文件夹，不自动生成制品
  • /opsx:continue：增量式地继续推进一个变更，一次只做一步
  • /opsx:ff：fast forward，一次性把一个空变更快速填满制品
  • /opsx:verify：在归档前做一次验证，检查规范和实现是否一致
  • /opsx:bulk-archive：批量归档多个变更
  • /opsx:onboard：给新接手项目的人或 AI 做一次引导

启用扩展工作流的方法：

大多数个人用户用 core profile 就够了，扩展工作流更适合团队协作或者复杂项目

与其他工具对比

vs Spec Kit

Spec Kit 是 GitHub 在 2025 年 9 月开源的 SDD 工具包

两者的定位很不一样：

  • Spec Kit：全面但偏重。有严格的阶段门禁，要写很多 Markdown，需要 Python 环境，初始配置大概要 30 分钟
  • Open Spec：轻量且自由。没有阶段门禁，可以随时回头改，不需要 Python，5 分钟就能上手

打个比方，Spec Kit 像是一套正式的西装，出席重要场合很合适，但平时穿太累；Open Spec 像是一件合身的卫衣，日常用着舒服，正式场合也撑得住

如果你是个人开发者或者小团队，Open Spec 几乎总是更顺手的选择。如果是大型企业、需要强流程管控的场景，Spec Kit 可能更合适

vs Kiro

Kiro 是 AWS 推出的 AI 编程 IDE，也内置了 SDD 能力

Open Spec 官方对 Kiro 的评价是：功能强大但被锁死在 Kiro 自己的 IDE 里，而且只能用 Claude 系列模型

而 Open Spec 的核心优势是工具无关——它支持 30 多种 AI 编程助手，包括 Claude Code、Cursor、GitHub Copilot、Windsurf、Gemini CLI、Codex 等等。你今天用 Cursor，明天换 Claude Code，规范文件原封不动地跟着走，不会被任何一家厂商绑死

这其实就是开放标准的价值。MCP 当年解决了 “AI 连接工具” 的标准化问题，Open Spec 想解决的是 “AI 读写规范” 的标准化问题

vs 无规范

最值得对比的其实是 “什么规范都不用” 的状态

不用任何规范框架时，你的需求只活在和 AI 的聊天记录里。问题是：

  • 聊天记录一长，AI 就忘了前面的约定
  • 换一次对话，所有上下文丢失
  • 同事想接手你的工作，没法看聊天记录
  • 半年后你想知道 “当时这个功能为什么这么设计”，无从查起

Open Spec 把这些散落在聊天里的需求固化成文件、纳入 Git 管理，让需求有了和代码同等的地位。它带来的不是更重的流程，而是更可靠的协作基础

实践与生态

既有项目改造

Open Spec 是 brownfield-first 的，这是它和很多同类工具最大的区别

在既有项目上用 Open Spec，不需要先把整个系统文档化。正确的做法是：

  1. 跑 openspec init 初始化目录结构
  2. 下次要改某个功能时，只为这次改动写一份 Delta Spec
  3. 改完归档，主 specs 里就多出了这个领域的规范
  4. 随着你不断做 Change，规范库自然覆盖到越来越多的领域

这种 “改到哪写到哪” 的策略，让既有项目的接入成本极低。你不需要停下来专门写文档，而是在正常开发的过程中顺手把规范建起来

Stores 跨仓库规划

当项目大到需要多个仓库协作时（比如一个功能要改 API 服务、Web 应用、共享库三个仓库），单个仓库里的 openspec/ 就不够用了

Open Spec 提出了 Stores 的概念：把规范放在一个独立的仓库里，用 git push 像共享代码一样共享给所有仓库和所有 AI 编程助手。这样：

  • 跨仓库的功能：一次变更、一份计划，代码落在三个仓库
  • 平台团队拥有规范，业务团队只读引用
  • 计划先于代码：先在 Store 里把计划定下来，各代码仓库再跟进实现

Stores 目前还是 beta 阶段，但思路很清晰——把规范当成和代码一样的一等公民来管理

支持的 AI 工具

Open Spec 不绑定任何特定 AI 工具，目前支持 30 多种主流 AI 编程助手，包括但不限于：

  • Claude Code
  • Cursor
  • GitHub Copilot
  • Windsurf
  • RooCode
  • Cline
  • Amazon Q
  • Codex
  • Gemini CLI

接入方式是 Open Spec 在初始化时会生成对应的 skills 和斜杠命令文件（比如在 .claude/skills/ 下放技能文件），支持的工具会自动识别这些文件，不需要你手动配置

模型选择建议

Open Spec 官方对模型选择有明确建议：推荐用高推理能力的模型，比如 Codex 5.5 和 Claude Opus 4.7，无论是规划阶段还是实现阶段都用它们

原因很简单：SDD 的核心价值在于 “想清楚再动手”，而 “想清楚” 这件事对模型的推理能力要求很高。用一个便宜的弱模型来省钱，结果 AI 在 propose 阶段就把需求理解错了，后面 apply 写出来的代码全是错的，反而更费钱

另外，Open Spec 强调上下文卫生（Context Hygiene）。每次开始 apply 之前清一下对话上下文，让 AI 专注于当前变更的制品，不要被无关历史干扰。这和前面在 Vibe Coding 概念里讲的上下文工程是一脉相承的思路

Open Spec 与 Vibe Coding 的关系

Open Spec 是 Vibe Coding 走向成熟的产物

Vibe Coding 是 “跟着感觉写代码”，Agentic Engineering 是 “按流程管 AI”。Open Spec 就是 Agentic Engineering 在 “需求管理” 这一环上的具体落地工具

Vibe Coding 时代，最大的瓶颈不是 AI 不会写代码，而是人不会提需求。Open Spec 通过一套结构化的规范文件，把 “提需求” 这件事从一门玄学变成了一个可重复、可审查、可归档的工程动作

可以说，谁掌握了用 Open Spec 这类工具把需求管好，谁就真正掌握了 AI 编程时代的核心竞争力。模型是商品，工具是商品，唯有 “把想法变成清晰规范” 的能力，是 AI 替代不了的
