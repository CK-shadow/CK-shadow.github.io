SDD 与 Spec Kit 基础

SDD 规范驱动开发

SDD（Spec-Driven Development，规范驱动开发）是 AI 时代的一种新型开发方法论，强调在编码之前先创建明确的、AI 能直接理解和执行的规范文档

传统开发流程是：想到什么写什么，边写边改，最后再补文档。这样容易导致需求不清晰、代码和文档对不上

而 SDD 的思路正好相反：先把需求写成规范文档，并且把规范文档当作代码的唯一真相来源

你可以把规范文档理解成 “项目宪法”，它包含了详细的需求描述、系统设计和接口定义。AI 必须严格遵守这些条文来生成代码，确保产出完全符合预期


Spec Kit 就是 GitHub 官方为落地 SDD 方法论而推出的一套开源工具包，也是目前 SDD 领域最有代表性的实现之一

Spec Kit 简介

Spec Kit 是 GitHub 在 2025 年 8 月开源的 SDD 工具包，托管在 github/spec-kit 仓库，采用 MIT 协议

它由 GitHub 官方团队维护，目前已经有 12 万多 Star、240 多位贡献者，是当前 Vibe Coding 领域最受关注的规范框架之一。和它定位类似的 Open Spec 走的是轻量自由路线，而 Spec Kit 走的是结构化、可扩展、面向企业的路线

Spec Kit 的核心思路可以用一句话概括：Define what to build before building it——在动手写代码之前，先定义清楚要做什么

它通过一套斜杠命令，把 SDD 的完整流程固化到你的 AI 编程助手里：先写规范（spec），再做计划（plan），再拆任务（tasks），最后才让 AI 去实现（implement）。每一步都会产出一个 Markdown 制品，喂给下一步当上下文，让你的 AI 拿到的是结构化的输入，而不是一句随口的提示词

打个比方，Vibe Coding 就像你口头跟实习生交代任务，说完就忘；Spec Kit 相当于给实习生发了一份正式的工单、设计图、施工清单，每一步都得按规矩来，谁都不能赖账

为什么需要 Spec Kit

AI 编程助手虽然强大，但 Vibe Coding 在真实项目上会稳定地翻车，而且翻车方式高度可预测

第一种是架构不一致。AI 读了你一部分代码，觉得某种模式合理，就照着做了，结果和你项目里其他地方的模式对不上，一个项目里出现了两套写法

第二种是脑补需求。你说 “加个 issue 类型支持”，AI 自己决定这是什么意思：固定列表还是动态拉取、批量选还是单条选、Jira 拒绝了怎么办。它猜得特别自信，但猜出来的边界情况谁也没规划过

第三种是没有共识。同事看你的 PR 只能看到代码，看不到代码背后的决策。”Epic” 是故意排除的吗？那个四项列表是业务决定还是临时原型？代码回答不了这些问题

第四种最贵——返工。实现完、Review 完，才发现做出来的根本不是 stakeholder 想要的，回头重做。而这些问题，本来花五分钟写个 spec 就能避免

Spec Kit 把 “写规范” 这件事从可有可无的文档工作，变成了开发流程的第一交付物。代码是实现细节，规范才是契约

Spec Kit 的设计哲学

Spec Kit 的设计理念可以归纳成几条非常清晰的原则：

  • Intent-driven 意图驱动：规范定义 “做什么”（what），计划定义 “怎么做”（how），二者严格分离
  • Multi-step refinement 多步精炼：不指望一次提示词就生成完美代码，而是通过多个阶段逐步打磨
  • Gate system 门禁系统：每个阶段都校验上一阶段的产出，没写 spec 就不能 plan，没 plan 就不能 tasks
  • Agent-agnostic 工具无关：支持 35+ 种 AI 编程助手，不绑死任何一家厂商
  • Extensible 可扩展：通过 presets、extensions、workflows、bundles 把流程拆成可组合的积木

其中最值得品味的是 Spec 与 Plan 的分离。很多团队写规范写着写着就变成了架构讨论，还没搞清楚用户要什么就开始争论用什么框架。Spec Kit 强制把这两件事拆开：spec 只描述用户价值，plan 才描述技术方案。这样产出的文档既不会让 stakeholder 看不懂，也不会让开发者觉得太模糊

另一条很关键的是 门禁系统。它不是官僚主义，而是和编译器拒绝链接未编译代码一个道理——每个阶段验证上一阶段的产出，过了才放行。这套机制让你不必靠记忆去维护流程纪律，工具替你守着

核心概念

Constitution 准则

Constitution（准则）是 Spec Kit 里项目级的 “宪法”，记录的是跨功能、跨特性都要遵守的根本原则：架构模式、技术选型、安全要求、约束条件

它和 spec 不一样：spec 是某一个功能的规范，constitution 是整个项目的底线。Constitution 只写一次，之后每个特性的 plan 阶段都会跑一次 Constitution Check，验证这次的设计没有违背这些根本原则

打个比方，Constitution 就像公司的章程，spec 就像某个项目的立项书。立项书可以一个接一个地写，但章程是大家都要遵守的底线。没有章程，每个项目就各自为政，架构慢慢就漂移了；有了章程，AI 每次做 plan 都会对照着检查，从源头杜绝架构走偏

Spec 规范

Spec（规范）是 Spec Kit 里描述 “要做什么” 的文档，用纯 Markdown 写成

一个 spec 的核心内容是用户价值，而不是技术方案。它包含：用户故事（user stories）、验收场景（acceptance scenarios）、边界情况（edge cases）、功能需求（functional requirements）、关键实体（key entities）、成功标准（success criteria）

spec 里不应该出现的东西：用什么框架、改哪些文件、数据模型长什么样。这些都属于 plan 的范畴。把 spec 和 plan 混在一起，会产出一个 stakeholder 看不懂、开发者又嫌太模糊的四不像

Spec Kit 还有一个好习惯：把假设（assumptions）当成一等公民写进 spec。比如 “只支持四种类型，不做动态拉取”、”Epic 必填字段按 best-effort 处理”。把这些写下来，它们就不会变成日后没人说得清的隐性约束

Plan 计划

Plan（计划）是 Spec Kit 里描述 “怎么做” 的技术文档，是 spec 之后的下一步

plan 阶段会读取 spec.md，深入研究代码库，然后产出完整的技术设计。它包含：gap analysis（差距分析，已有 vs 缺失）、data model（数据模型）、API contracts（接口契约）、implementation map（改动地图）、quickstart（快速验证步骤）

其中最有价值的产出是 gap analysis。它会清楚地告诉你：现有代码里已经有什么、还缺什么、这次改动要动几个文件、大约多少行代码。很多时候你会发现，基础设施其实已经在了，这次改动远比想象中小

plan 在生成前还会先跑一次 Constitution Check，确保技术方案没有违背项目根本原则。这一步是 Spec Kit 防止架构漂移的关键机制

Tasks 任务

Tasks（任务）是把 plan 拆解成可执行清单的产物，用 tasks.md 承载

Spec Kit 生成的任务列表有三个特点：

  • 扁平：不是嵌套树，而是一个平铺的列表，每个任务一个编号（T001、T002……）
  • 有序：按依赖关系排序，先做的排在前面，AI 可以从上往下依次执行
  • 独立可实施：每个任务都是一个可以独立完成、独立验证的小功能点

任务拆解的粒度很关键。太粗了 AI 容易跑偏，太细了又显得啰嗦。Spec Kit 的默认粒度是 “一个任务对应一个可独立验证的小功能点”

tasks 阶段有强门禁：如果你没写完 plan.md 就跑 tasks，它会直接报错拒绝。这就是门禁系统在工作

Implement 实现

Implement（实现）是让 AI 按 tasks.md 顺序写代码的阶段

这一步把控制权交给 AI 编程助手，同时把完整上下文一起喂过去：spec、plan、research、data model、API contracts、有序任务列表。AI 按顺序做任务，做完一项标记一项完成

因为所有决策都在前面阶段定好了，implement 阶段通常不会有惊喜。AI 不需要猜 “Epic 支持是什么意思”，不需要决定要不要加数据库迁移，不需要在批量选和单条选之间纠结——这些都已经在前面解决了。AI 只需要写代码

对于大型特性，implement 可以分阶段跑，一次只做一个 phase，避免一次塞太多上下文

Converge 收敛

Converge（收敛）是 Spec Kit 工作流的收尾命令，用来验证代码和规范是否真的对齐

它会把代码库和 spec/plan/tasks 三份制品做一次交叉比对。如果发现还有没实现的规范要求，它会把这些缺口作为新任务追加到 tasks.md 里。你接着跑一次 implement，再 converge，直到它报告 “已收敛”，就可以进入 review 或开 PR 了

converge 解决的是 “文档说做完了，代码其实没做完” 的脱节问题。它是整个 SDD 闭环的最后一道保险

质量门：Clarify / Checklist / Analyze

除了核心的五个命令，Spec Kit 完整工作流还提供三个质量门命令：

  • /speckit.clarify 澄清：在 plan 之前跑，针对 spec 里没说清楚的地方提出结构化问题，把你的答案折叠回 spec。它让你不会在模糊的需求上盖楼
  • /speckit.checklist 检查清单：在 plan 之后、tasks 之前跑，生成一份 “需求单元测试”，确认 spec 完整、清晰、一致。它把 “需求质量” 这件主观的事变成了可勾选的清单
  • /speckit.analyze 分析：在 tasks 之后、implement 之前跑，跨 spec.md、plan.md、tasks.md 做一致性分析，报告冲突、缺口、歧义。它是只读的，发现问题就回到源头改，改完再跑

这三个命令是 Spec Kit 比 Open Spec 更重的地方，也是它面向生产级特性的底气。短路径可以跳过它们，长路径（生产特性）建议都跑一遍

Gate System 门禁系统

门禁系统（Gate System）是 Spec Kit 区别于很多轻量工具的核心机制

它的规则很简单：每个阶段都校验上一阶段的产出，没过就不放行。具体来说：

  • 没 spec 就不能 plan
  • 没 plan 就不能 tasks
  • 没 tasks 就不能 implement
  • plan 阶段会跑 Constitution Check，违背准则就不过

这不是为了增加流程负担，而是为了让纪律变成工具的内置行为。你不必靠记忆去维护 “先想清楚再动手” 的习惯，工具替你守着。就像编译器不会让你链接没编译的代码一样，门禁系统从机制上杜绝了跳步

Spec vs Plan 分离

Spec 与 Plan 的分离，是 Spec Kit 工作流里最重要的一道分界线

spec 是用户价值文档：用 stakeholder 也能看懂的语言描述用户需要什么，不提技术。plan 是技术文档：精确到改哪些文件、数据模型长什么样、API 契约是什么

把它们合并会产生一个四不像文档：对 stakeholder 来说太技术，对开发者来说又太模糊。把它们分开，则各司其职——spec 让大家先就 “要什么” 达成一致，plan 再讨论 “怎么做”

这个分离还有一个隐性好处：当你回头审查一个老特性时，你能清楚地看到 “当时用户想要什么” 和 “当时我们决定怎么做”，两件事不会糊在一起

项目结构

.specify 目录

运行 specify init 之后，你的项目根目录会多出一个 .specify/ 目录，这是 Spec Kit 的 “大脑”，存放着脚本、模板、和 AI 编程助手的配置

其中 feature.json 是个很关键的设计。Spec Kit 通过它来追踪 “当前在做哪个特性”，而不是靠 Git 分支。也就是说，命令解析特性靠的是这个状态文件，而不是 git checkout。想切换到另一个特性，改这个文件（或设置 SPECIFY_FEATURE_DIRECTORY 环境变量）就行

这个设计的好处是：你甚至不需要 Git 就能用 Spec Kit。Git 是可选的扩展，开了之后会自动给特性加编号分支（比如 001-feature-name），但活跃特性仍然由 feature.json 决定

specs 目录

specs/ 是整个项目的真相来源，存放着每个特性的所有制品。每个特性一个独立子目录，按编号命名

这些文件会和代码一起提交到仓库里。它们回答了未来关于这个特性的所有问题：为什么这么设计、考虑过但没选的方案是什么、显式假设是什么。它们也是下一个基于此特性的功能开发时的输入

把规范放在代码仓库里、和代码一起用 Git 管理，是 Spec Kit 和 Open Spec 共享的设计。这样规范跟着代码走，不会出现 “代码在 GitHub，规范在某个文档平台，两边各活各的” 这种脱节

制品体系

Spec Kit 默认的制品有七类，每一类都有明确的职责：

制品 作用
spec.md 用户故事、验收场景、功能需求、成功标准（”做什么”）
plan.md 技术设计、差距分析、改动地图、Constitution Check（”怎么做”）
research.md 代码库研究记录，已有 vs 缺失
data-model.md 数据模型与实体定义
contracts/ API 契约变更，请求/响应 schema
quickstart.md 端到端手动验证场景
tasks.md 有序、独立可实施的任务列表

这些制品有严格的依赖关系：spec 决定 plan，plan 决定 tasks，tasks 驱动 implement。但这个依赖不是单向的水流——你在 implement 时发现 plan 要改，完全可以回头改 plan 甚至 spec，没有强制的阶段门禁会拦你（除了 “上一阶段必须存在” 这种基本门禁）

这种 “制品驱动 + 可回溯” 的设计，让 Spec Kit 既有结构又不失灵活

有个新手容易踩坑的点：specify 是在终端里跑的 CLI 命令，而 /speckit.* 是在 AI 助手对话框里输入的斜杠命令。两者别搞混，混了就跑不通。另外，不同 AI 工具的调用形式可能略有不同——基于 skills 的工具（如 Codex、ZCode）用 $speckit-*，Kimi 用 /skill:speckit-*，但步骤完全一样

/speckit.plan 是生成技术方案的命令，读取 spec.md，研究代码库，产出完整设计

它包含：技术栈选择、gap analysis、data model、API contracts、implementation map、quickstart。在生成前会先跑 Constitution Check

这一步是技术细节真正该出现的地方。spec 阶段不谈技术，plan 阶段才谈。这种分离让 spec 保持纯净，plan 可以放心讨论框架、文件、模式、迁移

plan 阶段最值钱的产出是 gap analysis。它清楚告诉你现有代码有什么、缺什么、这次要动几个文件、大约多少行。很多时候你会发现基础设施已经在了，改动远比想象中小

/speckit.checklist 检查清单

/speckit.checklist 是质量门命令，在 plan 之后、tasks 之前跑

它生成一份 “需求单元测试”——一份确认 spec 完整、清晰、一致的检查清单。把 “需求质量” 这件原本主观的事，变成了可勾选的清单

你可以把它理解成代码的单元测试，只不过测的是需求文档本身。在拆任务之前先确认需求没问题，比写完代码再发现需求漏洞便宜得多

/speckit.tasks 任务

/speckit.tasks 是把 plan 拆成任务列表的命令，读取所有设计制品，生成 tasks.md

它产出的任务列表有三个特点：扁平（不是嵌套树）、有序（按依赖排序）、独立可实施（每个任务可独立完成和验证）

这一步有强门禁：没写完 plan.md 就跑 tasks，会直接报错。这就是门禁系统在工作——它不让你跳步

/speckit.analyze 分析

/speckit.analyze 是质量门命令，在 tasks 之后、implement 之前跑

它跨 spec.md、plan.md、tasks.md 做一致性分析，报告冲突、缺口、歧义。它是只读的——发现问题就回到源头改，改完再跑

analyze 是 implement 前的最后一道关。它确保三份制品彼此对得上，避免 “spec 说一套、plan 说另一套、tasks 又是第三套” 的脱节

/speckit.implement 实现

/speckit.implement 是让 AI 按 tasks.md 顺序写代码的命令

它把完整上下文一起喂给 AI：spec、plan、research、data model、API contracts、有序任务列表。AI 按顺序做任务，做完一项标记一项完成

因为所有决策都在前面阶段定好了，implement 阶段通常不会有惊喜。AI 不需要猜需求，不需要做架构决策，只需要写代码。对于大型特性，可以分阶段跑，一次只做一个 phase

/speckit.converge 收敛

/speckit.converge 是收尾命令，验证代码和规范是否真的对齐

它把代码库和 spec/plan/tasks 三份制品做交叉比对。发现缺口就把新任务追加到 tasks.md，你接着跑 implement，再 converge，直到它报告 “已收敛”

converge 解决的是 “文档说做完了，代码其实没做完” 的脱节问题。它是 SDD 闭环的最后一道保险。收敛通过后，就可以进入 review 或开 PR 了

/speckit.taskstoissues 任务转 Issue

/speckit.taskstoissues 是个辅助命令，把生成的任务列表转成 GitHub Issues，方便在 GitHub 上追踪和执行

适合团队协作场景——任务不只是 AI 的待办，也是团队成员可以认领的工作项

扩展生态

Integrations 集成

Spec Kit 不绑定任何特定 AI 工具，目前官方支持 35+ 种主流 AI 编程助手，包括但不限于：

  • GitHub Copilot
  • Claude Code
  • Cursor
  • Codex
  • Gemini
  • Windsurf
  • Zed
  • Cline
  • Kilo Code
  • Forge
  • Kiro

接入方式是 specify init 时选择你的 AI 工具，Spec Kit 会自动生成对应的命令文件和目录结构。如果你的工具不在列表里，还有 generic 集成作为万能逃生口

切换 AI 工具只需要一条命令，规范文件原封不动地跟着走，不会被任何一家厂商绑死。这和 Open Spec 的工具无关理念是一致的——开放标准的价值在于不被锁死

Presets 预设

Presets（预设）是用来微调核心 SDD 流程的积木。Spec Kit 官方提供 25 个预设，可以针对不同技术栈、不同场景调整默认行为

比如 .NET 项目、Java 项目、前端项目可以各有各的 preset，让 plan 阶段生成的技术方案更贴合该栈的惯例。preset 不改变流程结构，只调整流程参数

Extensions 扩展

Extensions（扩展）是给核心流程加新能力的积木。目前社区有 138 个扩展，由 70+ 位作者贡献

典型扩展比如：

  • CI Guard：在 CI 里加合规门禁
  • Architecture Guard：加架构治理门禁

扩展可以让你在不 fork 整个 Spec Kit 的前提下，往流程里插入团队需要的额外检查

Workflows 工作流

Workflows（工作流）是用来编排多个步骤的积木。你可以把 specify → plan → tasks → implement 串成一个自动化的端到端流程，也可以拆成单步手动执行

对于小特性，跑自动化 workflow 一气呵成；对于复杂特性，逐步手动跑、每步审核更稳妥

Bundles 打包

Bundles（打包）是把 preset + extension + workflow 组合在一起、方便分享的机制。团队可以把一套定制流程打包成 bundle，在组织内分享，甚至发布到社区

Spec Kit 还支持企业自建 catalog，让组织内部统一管理推荐的集成、扩展、预设、workflow 和 bundle。这对企业落地非常重要——可以统一治理，而不是每个人各自装各自的

不止于 SDD

Spec Kit 最有意思的一点是：它的构建块足够通用，连 SDD 本身都可以被替换。社区已经基于它做出了完全不同的流程：

  • AIDE：7 步 AI 驱动的工程生命周期
  • Canon：基线驱动的工作流（spec-first、code-first、spec-drift）
  • Product Forge：面向产品管理的 SDD
  • FX→.NET：端到端的 .NET Framework 迁移，7 个阶段
  • MAQA：带质量保证门的多智能体编排
  • Fiction Book Writing：写长篇小说，从故事圣经到投稿

也就是说，Spec Kit 本质上是一个 “流程编排框架”，SDD 只是它默认搭载的一个流程。这种设计让它远远超出了一个 SDD 工具的范畴

与其他工具对比

vs Open Spec

Open Spec 是 Fission AI 团队在 2025 年 8 月开源的轻量级 SDD 框架，两者的定位很不一样：

  • Spec Kit：全面但偏重。有严格的阶段门禁，要写不少 Markdown，需要 Python 环境，初始配置大约 30 分钟，面向企业和生产级特性
  • Open Spec：轻量且自由。没有阶段门禁，可以随时回头改，不需要 Python，5 分钟就能上手，面向个人和小团队

打个比方，Spec Kit 像是一套正式的西装，出席重要场合很合适，但平时穿太累；Open Spec 像是一件合身的卫衣，日常用着舒服，正式场合也撑得住

如果你是个人开发者或者小团队，Open Spec 几乎总是更顺手的选择。如果是大型企业、需要强流程管控的场景，Spec Kit 可能更合适

一个很直观的对比：Spec Kit 有 9 个斜杠命令和门禁系统，Open Spec 只有 5 个斜杠命令且无门禁。重不重，看你的场景配不配得上这份重

vs Kiro

Kiro 是 AWS 推出的 AI 编程 IDE，也内置了 SDD 能力

Spec Kit 的核心优势是工具无关——它支持 35+ 种 AI 编程助手，你今天用 Copilot，明天换 Claude Code，规范文件原封不动地跟着走，不会被任何一家厂商绑死

而 Kiro 的 SDD 能力锁死在它自己的 IDE 里。功能强大但不自由

这其实就是开放标准的价值。MCP 当年解决了 “AI 连接工具” 的标准化问题，Spec Kit 和 Open Spec 想解决的是 “AI 读写规范” 的标准化问题

vs 无规范

最值得对比的其实是 “什么规范都不用” 的 Vibe Coding 状态

不用任何规范框架时，你的需求只活在和 AI 的聊天记录里。问题是：

  • 聊天记录一长，AI 就忘了前面的约定
  • 换一次对话，所有上下文丢失
  • 同事想接手你的工作，没法看聊天记录
  • 半年后你想知道 “当时这个功能为什么这么设计”，无从查起

Spec Kit 把这些散落在聊天里的需求固化成文件、纳入 Git 管理，让需求有了和代码同等的地位。它带来的不是更重的流程，而是更可靠的协作基础

适用场景

Spec Kit 不是所有场景都适合。判断标准是：如果你在 AI 开始实现前需要回答不止一个澄清问题，就先写 spec

适合用 Spec Kit 的场景：

  • 需求还没完全定义清楚的新功能
  • AI 需要做多个判断决策的模糊需求
  • 多人需要就 “要做什么” 达成一致的团队协作
  • 触及代码库多个部分的特性
  • 返工成本高的特性

不太适合的场景：

  • Bug 和修复都很明显的 hotfix
  • 行为不变的重构
  • 目标是发现需求而非实现需求的探索性原型
  • 一次性脚本，没有后续维护预期

简单说，越重要、越复杂、越多人协作的特性，越值得用 Spec Kit；越简单、越一次性的事，Vibe Coding 直接上反而更快

既有项目改造（Brownfield）

Spec Kit 也支持既有项目改造，官方把开发阶段分为三类：

  • 0-to-1 Development（Greenfield）：从零开始，先生成规范、再生成实现
  • Creative Exploration（创意探索）：并行探索多种实现方案、多种技术栈、多种 UX 模式
  • Iterative Enhancement（Brownfield）：在既有系统上迭代加功能、现代化遗留系统

对于既有项目，你不需要一开始就把整个系统文档化。正确的做法是：下次要改某个功能时，只为这次改动跑一遍 specify → plan → tasks → implement。改完之后，specs/ 里就多出了这个特性的完整规范。随着你不断做特性，规范库自然覆盖到越来越多的领域

这种 “改到哪写到哪” 的策略，让既有项目的接入成本可控。你不需要停下来专门写文档，而是在正常开发过程中顺手把规范建起来

模型选择建议

Spec Kit 官方推荐用高推理能力的模型，比如 Claude Opus 4.8、GPT-5.6 这类旗舰模型，无论是规范阶段还是实现阶段都用它们

原因很简单：SDD 的核心价值在于 “想清楚再动手”，而 “想清楚” 这件事对模型的推理能力要求很高。用一个便宜的弱模型来省钱，结果 AI 在 specify 阶段就把需求理解错了，后面 plan、tasks、implement 全是错的，反而更费钱

另外，Spec Kit 强调上下文卫生（Context Hygiene）。每次开始 implement 之前清一下对话上下文，让 AI 专注于当前特性的制品，不要被无关历史干扰。这和 Vibe Coding 概念里讲的上下文工程是一脉相承的思路

离线与企业部署

Spec Kit 对企业很友好：

  • 支持离线、防火墙后运行
  • 跨 Windows、macOS、Linux
  • 支持自建 catalog，统一治理组织内的集成、扩展、预设、workflow、bundle
  • 社区扩展如 CI Guard、Architecture Guard 提供合规门禁和治理能力

对于有合规要求、需要统一流程的企业，Spec Kit 的企业能力比 Open Spec 完善不少

Spec Kit 与 Vibe Coding 的关系

Spec Kit 是 Vibe Coding 走向成熟的产物

Vibe Coding 是 “跟着感觉写代码”，Agentic Engineering 是 “按流程管 AI”。Spec Kit 就是 Agentic Engineering 在 “需求管理” 这一环上的具体落地工具，而且是最重、最完整的那一个

Vibe Coding 时代，最大的瓶颈不是 AI 不会写代码，而是人不会提需求。Spec Kit 通过一套结构化的斜杠命令和门禁系统，把 “提需求” 这件事从一门玄学变成了一个可重复、可审查、可归档的工程动作

它和 Open Spec 解决的是同一个问题，只是取向不同：Open Spec 追求轻量自由，Spec Kit 追求结构严谨。前者像合身的卫衣，后者像正式的西装——选哪件，看你要去什么场合

可以说，谁掌握了用 Spec Kit 这类工具把需求管好，谁就真正掌握了 AI 编程时代的核心竞争力。模型是商品，工具是商品，唯有 “把想法变成清晰规范” 的能力，是 AI 替代不了的
